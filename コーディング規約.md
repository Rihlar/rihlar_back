## 1\. はじめに

本規約は、MSC (Model-Service-Controller) アーキテクチャを採用したプロジェクトにおける、一貫性のある保守しやすいコードベースの構築を目的とします。

-----

## 2\. MSC アーキテクチャの役割分担

プロジェクトでは、以下の役割分担を厳守してください。

  * **Model**:
      * データベースに関する処理（データの取得、保存、更新、削除など）を担当します。
      * SQL クエリや ORM (Object-Relational Mapping) の操作を集約します。
      * ビジネスロジックは含まず、純粋なデータ操作のみを行います。
  * **Service**:
      * アプリケーションの主要なビジネスロジックを担当します。
      * 複数の Model を組み合わせて複雑な処理を行う場合があります。
      * Controller から呼び出され、処理結果を Controller に返します。
  * **Controller**:
      * HTTP リクエストのハンドリングを担当します。
      * リクエストボディやクエリパラメータの取得、バリデーションを行います。
      * Service を呼び出し、その結果を基に HTTP レスポンス（JSON、HTML など）を整形して返します。
      * ビジネスロジックは含みません。

-----

## 3\. ユーザー ID の取得

-----

### 3.1 本番環境での取得方法

**ユーザー ID は、本来、認証ミドルウェアによって取得されるものとします。** 認証ミドルウェアがリクエストコンテキストやセッションに設定したユーザー ID を利用してください。

### 3.2 デバッグ時の取得方法

開発・デバッグ時は、一時的に HTTP ヘッダーからユーザー ID を取得することを許可します。このデバッグコードを記述する際は、以下の点に注意してください。

  * **必ず `// TODO` コメントを記述する:** ヘッダーからユーザー ID を取得するコードの上には、必ず `// TODO UserID の取得 (後々ミドルウェアからの取得に変更する)` のようにコメントを記述し、本番リリース前に削除・修正が必要であることを明示してください。

    ```go
    // TODO UserID の取得 (後々ミドルウェアからの取得に変更する)
    userid := ctx.Request().Header.Get("UserID")
    ```

## 4\. 命名規則とコメント

  * **命名規則**: 変数名、関数名、クラス名などは、プロジェクト全体で一貫した命名規則（**キャメルケース**または**パスカルケース**）に従ってください。
  * **コメント**: コードの意図や複雑なロジックには、簡潔かつ分かりやすいコメントを適宜記述してください。

## 5\. エラーハンドリング

  * エラーは適切に処理し、ユーザーには分かりやすいエラーメッセージを返却するようにしてください。

  * **Service レイヤーからのエラー返却**: Service レイヤーは、Go 言語標準の `error` 型ではなく、以下のカスタムエラー構造体を返すようにしてください。これにより、エラーに関する詳細な情報を統一的に扱えるようになります。

    ```go
    type CustomError struct {
        Code       int    // HTTPステータスコード
        LogMessage string // ログに出すメッセージ
        ErrMessage string // レスポンスに含めるエラーメッセージ
        Err        error  // 実際のエラー
    }
    ```

    カスタムエラー構造体がまだ定義されていない場合は、`error.go` ファイルで上記構造体を定義してください。

## 6\. テストデータの管理

テストデータは、以下の原則に従って管理してください。

  * **テストデータインサート関数の定義**: 新しく作成するテーブルの定義ファイル（例: `user_model.go`、`product_model.go`など）には、そのテーブルにテストデータをインサートするための専用の関数を定義してください。この関数は、テストに必要な初期データを投入する役割を担います。

      * テストデータインサート関数は、`Debug{テーブル名}`（例: `DebugSample()`, `DebugProfile()`）という命名規則で定義してください。

    <!-- end list -->

    ```go
    // 例: user_model.go 内
    func DebugUsers(db *gorm.DB) error {
        // テストユーザーをインサートするロジック
        // 例: db.Create(&User{Name: "Test User 1", ...})
        return nil
    }
    ```

  * **`models/debug.go`からの呼び出し**: 定義したテストデータインサート関数は、`models`ディレクトリ内の`debug.go`ファイルにある`Debug`関数から呼び出されるようにしてください。これにより、デバッグ時やテスト実行時に一括でテストデータを投入できるようになります。

    ```go
    // 例: models/debug.go 内
    func Debug() {
        // マイグレーション のコードをここに書く
        DeleteAndMigrate(Sample{})
        DeleteAndMigrate(Region{})
        DeleteAndMigrate(Member{})
        DeleteAndMigrate(Team{})
        DeleteAndMigrate(Game{})
        // DeleteAndMigrate(BaseChunk{})
        DeleteAndMigrate(GameChunk{})
        DeleteAndMigrate(Circle{})
        DeleteAndMigrate(MovementLog{})
        DeleteAndMigrate(Profile{})

        // デバッグのコードを呼び出す
        // DebugSample()

        DebugProfile()
        // DebugTeam()
        // DebugMember()

        // ゲームで使用するリージョンを作成する
        DebugRegion()

        // ゲームをデバッグする
        DebugGame()

        // DebugBaseChunk()
        // DebugGameChunk()

        // DebugPerformance()
        // DebugCircle()
        // DebugMovementLog()
    }
    ```

  * **`DeleteAndMigrate`関数の利用**: `DeleteAndMigrate`関数は、データベースのテーブルを一度すべてドロップし、その後、再度マイグレーションを実行する機能を提供します。この関数は、**テーブルのデータを完全に消去した状態でアプリケーションを使用したい場合**に利用してください。特に、開発環境やテスト環境でクリーンな状態からテストを開始したい場合に有効です。

    ```go
    // テーブルを削除してマイグレートする関数
    func DeleteAndMigrate(table interface{}) error {
        // テーブルを削除する
        err := dbconn.Migrator().DropTable(table)

        // エラー処理
        if err != nil {
            logger.PrintErr("テーブルの削除中にエラーが発生しました",err)
        }

        // マイグレーションする
        return dbconn.AutoMigrate(table)
    }
    ```

## 7\. フォルダ構成

プロジェクトのフォルダ構成は、以下の原則に従ってください。

  * **Service**: `services` ディレクトリ配下に配置してください。
  * **Model**: `models` ディレクトリ配下に配置してください。
  * **Controller**: `controllers` ディレクトリ配下に配置してください。
  * **Middleware**: `middlewares` ディレクトリ配下に配置してください。

-----

## 8\. ロギング

ロギングには、基本的に `logger` パッケージを使用してください。

  * **情報ログ (Info)**:
    ```go
    logger.Println("hello world")
    ```
  * **エラーログ (Error)**:
    ```go
    logger.PrintErr("hello world")
    ```
  * **例外的なロギング**: どうしても `logger` パッケージを使用できない場合は、独自のロギングメカニズムを使用しても構いません。
